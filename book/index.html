✨ C* Language Book

C* için ciddi, okunabilir ve yaşayan dil kitabı.

A serious, readable, living book for C*.

C*, C/C++ çizgisindeki düşük seviye kontrolü; açık ownership, module visibility, trait conformance, struct lifecycle ve LLVM backend ile birleştiren deneysel bir sistem programlama dilidir. Bu kitap hem bugün çalışan çekirdeği hem de proposal olarak tutulan vizyonu net sınırlarla anlatır.

C* is an experimental systems language that combines C/C++-style low-level control with explicit ownership, module visibility, trait conformance, struct lifecycle management, and an LLVM backend. This book separates what works today from the design proposals that shape the language.

✅ Güncel DurumCurrent Status

C* pre-alpha bir compiler projesidir. Yine de çekirdek artık sadece lexer/parser denemesi değil: LLVM IR üreten, executable çıkaran, smoke suite ile doğrulanan çalışan bir dil alt kümesi var. Aşağıdaki tablo, “bugün çalışıyor” ile “proposal” alanını karıştırmamak için özellikle keskin tutuldu.

C* is still a pre-alpha compiler project, but it is no longer only a parser experiment. It has a working subset that emits LLVM IR, links executables, and is validated by smoke tests. The table below deliberately separates implemented features from proposal surfaces.

🧭 Program Yapısı

C* blok tabanlıdır. Fonksiyon dönüş tipi :: ile yazılır, statement'lar genellikle ; ile biter, scope { ... } ile açılır. Yorumlar C/C++ tarzındadır.

C* is block based. Function return types use ::, statements usually end with ;, and scopes are delimited by { ... }. Comments are C/C++ style. 

// expected: pass
// expected-exit: 0

main() :: int32 {
  ret 0;
}
Ne oluyor?
  • main() :: int32 program girişidir; dönüş değeri process exit code olur.
  • ret 0; console'a yazı basmaz, generated executable'ın exit status değerini 0 yapar.
  • // expected etiketleri runner tarafından test niyeti olarak okunur.

🔢 Tipler, Literaller ve Cast

C* primitive tipleri açık genişlik taşır. Bu, ABI ve LLVM lowering tarafında niyeti net tutar. Numeric cast'ler iki yüzeye ayrılır: cast<T>(expr) ve expr as T. Pointer/int gibi tehlikeli dönüşümler için unsafe_cast kullanılır.

C* primitive types use explicit widths. Numeric casts are split between cast<T>(expr) and expr as T. Dangerous conversions such as integer-to-pointer use unsafe_cast. 

main() :: int32 {
  int8 tiny = 7;
  int32 whole = 42;
  uint64 mask = 255;
  float64 ratio = 0.5;
  char letter = 'A';
  bool ok = true;

  int32 from_float = cast<int32>(4.5);
  int32 also_cast = 5 as int32;

  if (ok && whole > tiny) {
    ret from_float + also_cast;
  }

  ret 0;
}
Örneğin anlattığı şey
  • Primitive tipler genişliklerini source üzerinde taşır; bu ABI ve LLVM storage seçimini açık tutar.
  • cast<int32>(4.5) ve 5 as int32 numeric conversion için iki çalışan yüzeydir.
  • ok && whole > tiny bool/logical expression lowering'ini gösterir.
Integer

int8..int64, int, isize, unsigned karşılıkları.

Floating

float32, float64, float; fractional literal lexer'da korunur.

C string

const char* string literal ve CRT interop için raw pointer ABI'de kalır.

🧩 Fonksiyonlar ve Parametre Semantiği

Fonksiyonlar source sırasından bağımsız çağrılabilir; compiler codegen öncesi prototipleri declare eder. Parametrelerde C*'a özgü bir ayrım var: bool y normal castable parametredir, x bool ise aynı primitive tipi taşır ama scope içinde implicit cast edilmesini yasaklar.

Functions can be called before their source definition because codegen declares prototypes first. A C*-specific parameter form exists: bool y is castable, while x bool carries the same primitive type but forbids implicit casts in the function scope. 

add(int32 a, int32 b) :: int32 {
  ret a + b;
}

castable(bool y) :: int32 {
  int32 value = y;
  ret value;
}

non_castable(x bool) :: bool {
  // int32 value = x; // diagnostic: Casting is not allowed for x
  ret x;
}

main() :: int32 {
  ret add(20, 22);
}
Bu örnek neden önemli?
  • add normal primitive parametreli fonksiyon çağrısını gösterir.
  • bool y cast edilebilir parametredir; gövdede int32 değişkene atanabilir.
  • x bool aynı primitive tipi taşır ama sembolün implicit cast edilmesini yasaklar.

🔀 Kontrol Akışı

if / elif / else expression değil statement bloklarıdır. loop(condition) while-style döngüdür. Range ve iterable formları da aynı loop yüzeyini kullanır.

if / elif / else are statement blocks. loop(condition) is a while-style loop. Range and iterable forms reuse the same loop surface. 

// expected-exit: 21

main() :: int32 {
  int32 i = 0;
  int32 total = 0;

  loop (i < 6) {
    if (i < 3) {
      total += 2;
    } else {
      total += 5;
    }

    i += 1;
  }

  ret total;
}
Kontrol akışı okuması
  • loop (i < 6) while-style döngüdür; condition her iterasyonda tekrar değerlendirilir.
  • if/else blokları total değerini iki farklı yoldan günceller.
  • i += 1 shortcut assignment codegen'ini ve loop çıkış koşulunu birlikte test eder.
main() :: int32 {
  int32 total = 0;
  int32 values[3] = (3, 4, 5);

  loop (i in [0, 3]) {
    total += i;
  }

  loop (value in values) {
    total += value;
  }

  loop (index, value in values) {
    total += index + value;
  }

  ret total;
}
Loop formları
  • loop (i in [0, 3]) numeric range loop'tur; loop değişkeni compiler tarafından local scope'a bağlanır.
  • loop (value in values) array iterable loop'tur; storage üzerinden element okunur.
  • loop (index, value in values) hem index hem element sembolünü loop scope'una ekler.

🧮 Diziler

C* dizileri tip yüzeyinde boyut taşır. Tek boyutlu parametreler, çok boyutlu sabit layout ve dynamic index smoke setinde doğrulanır. Iterable loop, array storage üzerinde GEP ile element okur.

C* arrays carry dimensions in the type surface. Single-dimensional array parameters, multi-dimensional fixed layouts, and dynamic indexes are covered by smoke tests. Iterable loops read elements through GEP. 

read_mid(int32[2:3] matrix) :: int32 {
  ret matrix[1:1];
}

write_first(int32[2] values) :: void {
  values[0] = 9;
  ret;
}

main() :: int32 {
  int32 values[2] = (4, 5);
  int32 matrix[2:3] = (
    (1, 2, 3),
    (4, 5, 6)
  );

  write_first(values);
  ret values[0] + read_mid(matrix);
}

🏷️ Scalar Enum

Scalar enum, C ABI'ye yakın açık repr modelidir. Enum inheritance yapmaz, payload taşımaz ve struct layout üretmez. Member erişimi canonical olarak Color.Green biçimindedir. Underlying integer storage codegen tarafında doğrudan kullanılır.

Scalar enum is an explicit-representation model close to C ABI enums. It does not inherit, carry payloads, or emit a struct layout. Member access is canonical as Color.Green, and codegen uses the underlying integer storage. 

enum Color : uint8 {
  Red,
  Green,
  Blue = 7,
}

choose(Color color) :: Color {
  ret color;
}

main() :: int32 {
  Color color = Color.Green;
  Color selected = choose(color);
  ret 0;
}
Enum semantiği
  • : uint8 explicit repr'dır; scalar enum storage'ı bu integer tipe iner.
  • Color.Green canonical qualified member erişimidir; unqualified Green açılmadı.
  • choose(Color color) :: Color enum parametre/return type-check hattını doğrular.

Henüz yok: `flags enum`, repr overflow diagnostic hardening, exhaustiveness ve `tagged` payload layout proposal aşamasındadır. Scalar enum MVP gerçek compiler hattındadır.

Not yet: `flags enum`, representation overflow hardening, exhaustiveness, and `tagged` payload layouts remain proposals. Scalar enum is implemented.

🔐 Pointer, Reference ve Ownership

C* pointer modelinde sembol, sahiplik politikasını tipin üstünde taşır. * shared handle, ^ unique handle, & reference type, ref address/reference expression, deref ise dereference expression'dır. Bu model raw pointer ile shared ownership'ü aynı syntax'a sıkıştırmamak için bilinçli olarak ayrıdır.

In C*, pointer syntax carries ownership policy. * is a shared handle, ^ is a unique handle, & is a reference type, ref forms an address/reference expression, and deref dereferences. This deliberately avoids hiding ownership behind raw pointer syntax. 

read_ptr(int32* pointer) :: int32 {
  ret deref pointer;
}

take_unique(nomove int32^ pointer) :: int32 {
  ret deref pointer;
}

main() :: int32 {
  int32 value = 7;
  int32* shared = ref value;
  int32* copy = shared;

  int32^ owned = ref value;
  int32 result = take_unique(owned);

  ret read_ptr(copy) + result + cast<int32>(strong_count(copy));
}
Ownership okuması
  • int32* shared = ref value shared handle üretir; copy = shared retain/copy davranışıdır.
  • int32^ owned unique handle'dır; copy yerine explicit transfer gerekir.
  • nomove parametre, callee içinde unique pointer'ın tekrar taşınmasını engeller.
YüzeySemantikTipik diagnostic

T^Unique handle; copy yasak, transfer := veya move ile açık.Unique pointer cannot be copied.
nomove T^Callee içinde yeniden move edilemeyen unique parametre.nomove pointer cannot be moved.

🏗️ Struct Lifecycle

C* struct modeli C++ constructor/free karmaşasını bilerek ayırır: constructor allocation değildir, destructor free değildir. Constructor mevcut storage'ı initialize eder; new Type(args) allocation ve constructor çağrısını compiler lowering olarak birleştirir. drop explicit erken release'tir.

C* deliberately separates construction from allocation: constructors do not allocate, destructors do not mean free by themselves. Constructors initialize existing storage; new Type(args) combines allocation and construction as compiler lowering. drop performs explicit early release. 

struct Meter {
  int32 value;

  constructor(int32 initial) {
    self.value = initial;
    ret;
  }

  destructor() {
    self.value = 0;
    ret;
  }
}

main() :: int32 {
  Meter^ meter = new Meter(9);
  int32 value = meter.value;
  drop meter;
  ret value;
}

Method, self ve static

struct Counter {
  int32 value;

  constructor(int32 value) {
    self.value = value;
  }

  increment() :: void {
    self.value += 1;
    ret;
  }

  read :: int32 {
    ret self.value;
  }

  static zero() :: Counter {
    ret Counter(0);
  }
}

🧬 Trait ve Value Operator

Trait static conformance modelidir: runtime vtable üretmez, data inheritance yapmaz. Struct bir trait'i with ile açıkça üstlenir; required method eksikse pass1 diagnostic üretir. Value operator'lar struct method ABI'siyle çalışır; lifecycle/allocation operator'ları compiler-reserved kalır.

Traits are static conformance contracts: no runtime vtable, no data inheritance. A struct explicitly declares with Trait; missing required methods are diagnosed. Value operators lower through the struct method ABI, while lifecycle/allocation operators are compiler-reserved. 

trait Measurable {
  value() :: int32;
}

struct Meter with Measurable {
  int32 reading;

  constructor(int32 value) {
    self.reading = value;
  }

  value() :: int32 {
    ret self.reading;
  }

  operator +(Meter rhs) :: Meter {
    ret Meter(self.reading + rhs.reading);
  }

  operator ==(Meter rhs) :: bool {
    ret self.reading == rhs.reading;
  }
}

main() :: int32 {
  Meter a = Meter(3);
  Meter b = Meter(4);
  Meter c = a + b;
  ret c.value();
}

📦 Modül, Visibility ve Native Interop

include "file.cstar" as alias source-level public declaration merge yapar. Modifier yazılmayan declaration private kabul edilir. import ve export visibility değil, native/linkage ABI yüzeyidir.

include "file.cstar" as alias performs a source-level public declaration merge. Declarations without a modifier are private. import and export are native/linkage ABI surfaces, not visibility modifiers. 

// modules/public_static_module.cstar
public static limit :: int32 = 6;

public double_limit() :: int32 {
  ret limit * 2;
}

// main.cstar
include "modules/public_static_module.cstar" as cfg

main() :: int32 {
  ret cfg.double_limit();
}
import from "std:crt" {
  abs(int32 value) :: int32;
  printf(const char* fmt, ...) :: int32;
}

main() :: int32 {
  int32 value = abs(-42);
  printf("value=%d\n", value);
  ret value;
}

📚 Stdlib MVP ve Console Runtime

İlk standart kütüphane yüzeyi std/core.cstar dosyasındadır. Bu dosya CRT import üstünden core_print* ve core_read* wrapper'larını sağlar. Bunun yanında compiler runtime tarafında geçici console builtin'leri de vardır: print, input_int, input_string, clear_screen, flush_output, sleep_ms, enable_raw_input, disable_raw_input, read_key.

The first standard library surface lives in std/core.cstar. It wraps CRT imports with core_print* and core_read*. Temporary console builtins also exist for demos. 

include "../../std/core.cstar" as core

main() :: int32 {
  core.core_print("name: ");
  char* name = core.core_read_string();

  core.core_print("score: ");
  int32 score = core.core_read_i32();

  core.core_print(name);
  core.core_print(" -> ");
  core.core_print_i32(score);
  core.core_print("\n");

  ret 0;
}
Stdlib kullanım akışı
  • std/core.cstar CRT import'larını C* wrapper fonksiyonlarına bağlayan ilk stdlib MVP'sidir.
  • core_read_string() geçici static buffer döndürür; gerçek string modeli gelince değişecek nokta burasıdır.
  • core_print_i32 ve benzeri wrapper'lar sayısal output için printf formatlarını saklar.

🩺 Diagnostic Modeli

Diagnostic çıktısı dosya yolu, satır, sütun, severity, hata kodu ve caret marker taşır. Runner, // expected-code: CSTNNNN etiketini okuyup beklenen diagnostic kodunun gerçekten geldiğini doğrular. Crash/assert kabul edilmez; bilinçli hatalar controlled diagnostic üretmelidir.

Diagnostics include file path, line, column, severity, code, and a caret marker. The runner validates // expected-code: CSTNNNN tags. Crashes/asserts are not acceptable; intentional failures must produce controlled diagnostics. 

// expected: diagnostic
// expected-code: CST2001

enum Color : uint8 {
  Red,
  Green,
}

main() :: int32 {
  Color color = Color.Blue; // Enum 'Color' has no member 'Blue'
  ret 0;
}
CST1001

Parser/proposal grammar hataları: unexpected token, proposal keyword veya reserved operator kullanımı.

CST2001

Semantic/type-check hataları: symbol lookup, incompatible type, enum member, trait conformance.

CST2100

Qualifier mismatch: const/readonly/constref/constptr kuralları.

CST2105

Ownership/lifecycle: moved/dropped value use, direct destructor call.

📖 Derin Referans: C* Nasıl Okunur?

Bu bölüm kısa özellik listesi değil, dili gerçekten kullanırken akılda tutulması gereken modeldir. C* şu anda pre-alpha olsa da artık “birkaç token parse eden” proje değil; lexical analysis, AST, symbol pass, type-check pass, LLVM IR üretimi, native linkleme ve örnek runner zinciri olan küçük bir compiler hattı var. Bu yüzden dokümantasyonu da iki ayrı renkte okumak gerekir: MVP dediğimiz şey smoke/type_checker ile yaşayan compiler davranışıdır; proposal dediğimiz şey ise `examples/papers/` altında dili büyütmek için tutulan tasarım yüzeyidir.

1. Derleme Hattı: Kaynaktan Exe'ye

Bir `.cstar` dosyası çalıştırıldığında compiler tek seferde “kaynağı oku ve exe üret” yapmıyor; her biri ayrı sorumluluk taşıyan pass'lerden geçiyor. Bu ayrım hata mesajlarının doğru yerde üretilmesi için önemli. Örneğin `func<A>()` gibi template'e benzeyen ama henüz dilde olmayan bir yüzey parser diagnostic'i üretirken, `Color.Blue` gibi var olmayan enum üyesi pass1 semantic diagnostic'i üretir.

  1. CLI ve source load: `main.cpp` dosya yolunu, `--run`, `--stats`, `--verbose`, output klasörü ve backend ayarlarını okur. Normal kullanımda sadece banner görünür; pass süreleri `--stats` ile açılır.
  2. Lexical analysis: kaynak metin token stream'e çevrilir. Keyword, identifier, literal, operator, newline continuation ve yorum ayrımı burada yapılır.
  3. Syntactic analysis: parser token stream'den AST üretir. Bloklar, declaration'lar, expression precedence, function body, struct/enum yüzeyleri burada şekillenir.
  4. Pass 0 / Symbol analysis: scope, module alias, function/struct/enum sembolleri, redefinition ve visibility gibi isim dünyası kuralları doğrulanır.
  5. Pass 1 / Type checking: expression type, cast, qualifier, ownership flow, enum member, trait conformance, function call ve return kuralları doğrulanır.
  6. Codegen: AST LLVM IR'e indirilir. `.ll`, assembly/object ve executable varsayılan olarak `.cstar-out/` altına yazılır; root klasör kirletilmez.
  7. Backend/link/run: executable üretimi için LLVM IR Clang driver'a verilir. `--run` varsa üretilen program çalıştırılır ve process exit code runner tarafından yakalanır.
.\build-ucrt\cstar.exe .\examples\smoke\minimal.cstar

  • --stats pass süreleri ve LoC gibi compiler içi ölçümleri gösterir.
  • --run üretilen executable'ı çalıştırır; `ret N` console output değil process exit code'dur.
  • Runner, `// expected: pass`, `// expected: diagnostic`, `// expected-exit: N` ve `// expected-code: CSTNNNN` etiketlerini test sözleşmesi olarak kullanır.

2. Programın Ana Şekli

C* dosyası top-level declaration'lardan oluşur. Function, global variable, struct, trait, enum, include ve native import/export top-level olabilir. Program entry point'i klasik olarak main() :: int32 fonksiyonudur. Return type için :: kullanılır; bu, declaration ile type alanını gözle görünür biçimde ayırır. 

// expected: pass
// expected-exit: 9

public static scale :: int32 = 3;

double(int32 value) :: int32 {
  ret value * 2;
}

main() :: int32 {
  int32 local = double(scale);
  ret local + 3;
}
Satır satır okuma
  • public static scale :: int32 = 3; module dışından görülebilen static storage'dır.
  • double(int32 value) :: int32 parametre tipi solda, dönüş tipi :: sonrasında yazılır.
  • ret local + 3; process exit code'u üretir; terminale yazı basmaz.
  • Bu örnek `expected-exit: 9` ile runner tarafından davranış olarak doğrulanabilir.

3. Tip Sistemi: Scalar, Pointer, Struct, Enum

C* tip sistemi “her şeyi object” yapmaz. Primitive scalar tipler doğrudan LLVM scalar/storage'a iner; enum explicit integer representation taşır; struct alanları değer layout'u üretir; pointer/ownership marker'ları ise type category'nin parçasıdır. Bu ayrım C ABI ve sistem programlama hedefi için bilinçli.

int32

32-bit signed integer. Return code, loop counter ve arithmetic smoke'larında ana örnek tip.

float32 / float64

Fractional literal desteğiyle LLVM floating arithmetic üretir.

bool

Condition conversion ve logical expression hattında kullanılır; integer'a implicit cast kuralları type-check edilir.

char* / const char*

String literal ve CRT interop tarafında raw C pointer ABI olarak korunur, shared pointer retain gibi davranmaz.

T*

Shared handle semantics. Copy/assignment retain-release sayacı ve `strong_count` ile test edilir.

T^

Unique handle semantics. Copy yasak, ownership transfer `move` veya `:=` ile açık yapılır.

// expected: pass
// expected-exit: 42

enum Mode : uint8 {
  Idle,
  Active = 2,
}

score(bool enabled, int32 base) :: int32 {
  if (enabled && base > 10) {
    ret base * 3;
  }
  ret base;
}

main() :: int32 {
  Mode mode = Mode.Active;
  int32 result = score(true, 14);
  ret result;
}
Type-check burada neyi kanıtlıyor?
  • Mode scalar enum'dur; storage tarafında uint8 repr kullanır.
  • enabled && base > 10 logical expression içinde bool condition beklenir.
  • base * 3 integer arithmetic üretir ve dönüş tipi int32 ile uyumludur.
  • mode kullanılmasa da enum local declaration ve member lookup pass1 hattından geçer.

Parametrede sıra önemli: C* proposal'ında bool y ile x bool aynı tip bilgisini taşısa bile aynı semantic niyeti taşımaz. bool y normal cast edilebilir primitive parametre gibi davranırken, x bool sembolün scope içinde implicit cast edilmesini yasaklayan “non-castable parameter” kuralını temsil eder. Bu artık smoke ve type_checker örnekleriyle belgelenmiş bir davranıştır.

4. Expression, Assignment ve Cast Modeli

C* expression parser'ı arithmetic, comparison, logical, function call, field access, subscript ve cast yüzeylerini tek precedence hattında toplar. Assignment statement düz storage write olarak okunur; shortcut assignment önce mevcut değeri okur, sonra operator uygular, sonra storage'a geri yazar. 

// expected: pass
// expected-exit: 12

main() :: int32 {
  int32 value = 5;
  value += 3;

  float64 ratio = 4.5;
  int32 rounded = cast<int32>(ratio);
  int32 same = rounded as int32;

  ret value + same;
}
YüzeyAnlamDiagnostic sınırı
cast<T>(expr)Explicit, type-check edilen dönüşüm.Unsafe kategori değişimleri veya user-defined cast henüz controlled diagnostic olabilir.
expr as TOkunabilir safe cast sugar'ı.Qualifier stripping ve pointer/value kategori geçişi reddedilir.
unsafe_cast<T>(expr)Sistem programlama için bilinçli tehlikeli dönüşüm.Integer to pointer MVP var; sorumluluk caller'dadır.
+=, -=, *=Read-modify-write assignment.Const/readonly/invalid storage yazımı diagnostic üretir.
Neden iki cast yüzeyi var?
  • cast<T> compiler'a “bu dönüşümü bilinçli istiyorum” der, ama hâlâ safe kategoridedir.
  • as aynı niyeti daha okunabilir expression içine yerleştirir.
  • unsafe_cast<T> ise C tarzı low-level escape hatch'tir; dokümanda özellikle ayrı tutulmalıdır.

5. Fonksiyonlar: ABI, Parametre ve Return

Fonksiyon call hattı sadece primitive argüman geçirmekten ibaret değil. Bugünkü smoke seti symbol argument, cast edilmiş argument, pointer argument, struct by-value parametre/return, forward call ve void call statement yüzeylerini kapsıyor. C* için önemli kural şudur: parametre tipi call boundary'de hem storage kategorisini hem ownership niyetini taşır. 

// expected: pass
// expected-exit: 19

read_value(int32* pointer) :: int32 {
  ret deref pointer;
}

write_value(int32& output, int32 value) :: void {
  output = value;
  ret;
}

mix(int32 left, int32 right) :: int32 {
  ret left * 2 + right;
}

main() :: int32 {
  int32 value = 3;
  write_value(ref value, 7);
  ret mix(read_value(ref value), 5);
}
Call boundary nasıl okunur?
  • read_value(int32* pointer) shared/raw pointer kategorisiyle bir adres okur.
  • write_value(int32& output, ...) reference parametre bekler; caller ref value vermelidir.
  • void fonksiyonlarda ret; kullanılabilir; bu statement codegen'de boş return üretir.
  • Yanlışlıkla çıplak value reference parametreye verilirse type_checker controlled diagnostic üretir.

6. Array Modeli: Storage, Parametre ve Iterable Loop

Array'ler bugünkü çekirdekte hem tek boyutlu hem çok boyutlu storage olarak çalışır. Çok boyutlu subscript C* yüzeyinde matrix[row:col] şeklindedir; bu model syntax olarak C'nin matrix[row][col] zincirinden daha açık bir “multi-dimensional index tuple” hissi verir. Parametrede array boyutu type'ın parçasıdır; int32[2] ile int32[3] aynı tip değildir. 

// expected: pass
// expected-exit: 27

write_first(int32[3] values, int32 value) :: void {
  values[0] = value;
  ret;
}

sum_matrix(int32[2:3] matrix) :: int32 {
  int32 total = 0;
  loop (row in [0, 1]) {
    loop (col in [0, 2]) {
      total += matrix[row:col];
    }
  }
  ret total;
}

main() :: int32 {
  int32 values[3] = (1, 2, 3);
  int32 matrix[2:3] = (
    (1, 2, 3),
    (4, 5, 6)
  );

  write_first(values, 6);
  ret values[0] + sum_matrix(matrix);
}
Array örneği neyi gösteriyor?
  • int32[3] parametre, caller'dan aynı boyutta array ister.
  • matrix[row:col] dynamic index ile çok boyutlu element read yapar.
  • Range loop'lar inclusive aralık smoke'larıyla test edilir; nested loop codegen branch/phi düzenini de sınar.
  • Array parametreye scalar veya farklı boyutlu array geçirmek type_checker diagnostic üretir.

7. Kontrol Akışı: If, Loop, Break, Continue

C* kontrol akışı C-benzeri fakat tek loop keyword'lüdür. loop (condition) while gibi, loop (i in [min, max]) range loop gibi, loop (value in array) iterable loop gibi okunur. Bu tasarım, “for/while/foreach” ayrımını syntax yerine pattern'e taşır. 

// expected: pass
// expected-exit: 8

main() :: int32 {
  int32 total = 0;

  loop (i in [0, 5]) {
    if (i == 2) {
      continue;
    }

    if (i == 5) {
      break;
    }

    total += i;
  }

  ret total;
}
Branch davranışı
  • continue loop update/next iteration noktasına gider.
  • break loop çıkış block'una gider.
  • Loop dışındaki break/continue kullanımı parser başarısı değil semantic diagnostic'tir.
  • Pointer ve scalar condition conversion ayrı smoke dosyalarıyla doğrulanır.

8. Ownership: Shared, Unique, Ref, Drop

C*'ın sistem dili karakteri en çok pointer/ownership modelinde görünür. * shared handle, ^ unique handle, & reference parametre/ tip, ref adres alma, deref dereference ve drop lifecycle bitirme yüzeyidir. Bu modelin hedefi, C++'taki implicit copy/lifetime sürprizlerini azaltmak ve maliyeti syntax'ta görünür tutmaktır. 

// expected: pass
// expected-exit: 31

consume(nomove int32^ pointer) :: int32 {
  ret deref pointer;
}

main() :: int32 {
  int32 value = 31;

  int32* shared = ref value;
  int32* copied = shared;

  int32^ unique = ref value;
  int32 result = consume(unique);

  ret result + cast<int32>(strong_count(copied)) - 2;
}

9. Struct Lifecycle: Constructor Allocation Değildir

Struct proposal'ının en önemli kararı şudur: constructor allocation değildir, destructor free değildir. Constructor mevcut storage'ı initialize eder. new Type(args) compiler-reserved allocation operator'ıdır ve kullanıcı tarafından method adı gibi tanımlanmaz. Customization gerekiyorsa allocator trait/arayüzü üzerinden yapılır; C++'taki user-defined `operator new` karmaşasına girilmez. 

// expected: pass
// expected-exit: 12

struct Point {
  int32 x;
  int32 y;

  constructor(int32 x, int32 y) {
    self.x = x;
    self.y = y;
    ret;
  }

  sum :: int32 {
    ret self.x + self.y;
  }

  destructor() {
    self.x = 0;
    self.y = 0;
    ret;
  }
}

main() :: int32 {
  Point value = Point(5, 7);
  Point^ heap = new Point(1, 2);

  int32 result = value.sum() + heap.sum();
  drop heap;

  ret result - 3;
}
Lifecycle okuması
  • Point value = Point(5, 7) stack/local storage üstünde constructor initializer'dır.
  • new Point(1, 2) heap allocation + constructor lowering yapar.
  • drop heap explicit lifecycle bitişidir; sonrasında heap kullanılamaz.
  • By-value local struct scope çıkışında destructor lowering smoke ile doğrulanır.

10. Trait ve Operator: Static Conformance

Trait sistemi bugün static conformance modelindedir. Bu Rust'taki dynamic trait object modelinin aynısı değildir; C* burada data inheritance da yapmaz. Struct bir trait'i with ile üstlenir ve required method'ları sağlamazsa compiler diagnostic üretir. Dynamic trait proposal'ı ayrı dosyada tutulur ve explicit dynamic Trait&, dynamic Trait*, dynamic Trait^ gibi maliyeti görünür yüzeylerle büyütülmelidir. 

// expected: pass
// expected-exit: 13

trait Measurable {
  value() :: int32;
}

struct Meter with Measurable {
  int32 reading;

  constructor(int32 reading) {
    self.reading = reading;
  }

  value() :: int32 {
    ret self.reading;
  }

  operator +(Meter rhs) :: Meter {
    ret Meter(self.reading + rhs.reading);
  }
}

main() :: int32 {
  Meter a = Meter(6);
  Meter b = Meter(7);
  Meter c = a + b;
  ret c.value();
}

Okuma kuralı: `with Measurable` runtime polymorphism sözü değildir; “bu struct bu compile-time sözleşmeyi sağlıyor” demektir. Dynamic dispatch gerektiğinde syntax'ta `dynamic` kelimesi görünmelidir. Bu, sistem dili tasarımında maliyeti saklamamak için bilinçli bir ayrımdır.

11. Module ve Native Interop: Kaynak Yüzeyi vs Linker Yüzeyi

C* iki farklı dış dünya bağlantısını ayırır. include C* kaynak seviyesinde module yüzeyidir; alias ile public declaration'lara erişilir. import/export ise native/linker yüzeyidir; CRT, platform library veya dış ABI ile konuşur. Bu ayrım özellikle stdlib büyürken önemlidir: stdlib C* wrapper verebilir ama en altta CRT import kullanabilir. 

// examples/smoke/modules/math_module.cstar
public add_from_module(int32 a, int32 b) :: int32 {
  ret a + b;
}

// app.cstar
include "modules/math_module.cstar" as math

import from "std:crt" {
  printf(const char* fmt, ...) :: int32;
}

main() :: int32 {
  int32 value = math.add_from_module(20, 22);
  printf("value=%d\n", value);
  ret value;
}
İki ayrı boundary
  • math.add_from_module C* module alias lookup'tur; visibility kuralları pass0/pass1'de kontrol edilir.
  • printf native import'tur; vararg kısmı C ABI sorumluluğu taşır.
  • const char* string literal ABI için raw pointer olarak geçer; shared ownership handle değildir.

12. Diagnostics ve Negatif Test Yazma

C* için başarısız örnekler de değerli testtir. Bir syntax veya semantic kuralı reddedilecekse, bunun crash/assert değil controlled diagnostic üretmesi gerekir. `examples/type_checker` bu yüzden “kırık kod” klasörü değil; compiler davranış sözleşmesidir. 

// expected: diagnostic
// expected-code: CST2001

enum Color : uint8 {
  Red,
  Green,
}

main() :: int32 {
  Color color = Color.Blue;
  ret 0;
}
Negatif örnek nasıl okunur?
  • // expected: diagnostic bu dosyanın geçmesini değil kontrollü hata üretmesini bekler.
  • // expected-code: CST2001 sadece herhangi hata değil, belirli diagnostic ailesini bekler.
  • Runner crash/assert görürse bu “beklenen diagnostic” sayılmaz; compiler'ın sağlamlığı açısından başarısızlıktır.
  • Bu model dili büyütürken regression yakalamak için küçük ama çok etkili bir sözleşmedir.

`examples/papers/*.cstar` dosyaları çalışan suite gibi okunmamalıdır. Bunlar syntax ve semantic vizyonu göstermek için yazılmış proposal belgeleridir. `struct.cstar`, `allocator.cstar`, `trait.cstar`, `protocol.cstar`, `enum.cstar`, `metaprogramming.cstar`, `concurrency.cstar` gibi dosyaların amacı, compiler'a eklenecek özellikleri gerçek C* koduna benzeyen örneklerle tartışmaktır.

🗺️ Proposal ve Yol Haritası

C* tasarımı tek bir gizli hook sistemine her şeyi yüklememeye karar verdi. Allocation, protocol state, trait dispatch, cleanup, error flow, macro expansion ve async boundary ayrı kavramlar olarak büyütülecek. Bu dilin sistem programlama iddiası için iyi bir disiplin: runtime maliyeti varsa syntax'ta görünmeli.

C* deliberately avoids putting every concern under one hidden hook system. Allocation, protocol state, trait dispatch, cleanup, error flow, macro expansion, and async boundaries will grow as separate concepts. For a systems language, runtime cost should be visible in syntax.

🏷️ Enum hardening

Flags enum, repr overflow diagnostic, duplicate value policy, option exhaustiveness.

🧾 Macro / Attribute

Typed AST substitution, hygienic macro, reflection metadata, `#if/#error/#warning`.

🔁 Protocol

Static typestate flow, `scope_exit`, dynamic protocol `.=` desugar, no hidden hook table.

🧬 Dynamic Trait

`dynamic Trait&/*/^`, explicit type erasure, vtable layout contract.

⚠️ Effects

`except`, `throw`, `defer`, cleanup safety and structured early-return lowering.

⚙️ Async

Task ownership, `Send`/`Sync`, `nomove`, structured concurrency and scheduler model.

Sıradaki sağlıklı compiler işi: scalar enum'dan sonra flags enum ve repr overflow hardening, ardından tagged layout veya protocol flow analysis. Tagged access ve protocol state proof aynı statik kanıtlama altyapısını paylaşabilir.

Next healthy compiler work: after scalar enums, implement flags enum and representation overflow hardening; then move to tagged layout or protocol flow analysis. Tagged access and protocol state proof can share the same static proof infrastructure.

Mini module + enum + struct

enum Level : uint8 {
  Low,
  Medium,
  High = 10,
}

trait Renderable {
  render() :: int32;
}

struct Gauge with Renderable {
  Level level;
  int32 value;

  constructor(Level level, int32 value) {
    self.level = level;
    self.value = value;
  }

  render() :: int32 {
    if (self.level == Level.High) {
      ret self.value * 2;
    }
    ret self.value;
  }
}

main() :: int32 {
  Gauge gauge = Gauge(Level.High, 21);
  ret gauge.render();
}

Pointer ownership handoff

struct Resource {
  int32 value;
  constructor(int32 value) { self.value = value; }
  destructor() { self.value = 0; }
}

consume(Resource^ resource) :: int32 {
  int32 value = resource.value;
  drop resource;
  ret value;
}

main() :: int32 {
  Resource^ resource = new Resource(33);
  ret consume(move resource);
}
