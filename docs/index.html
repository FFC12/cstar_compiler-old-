§01

Giriş ve Dil FelsefesiIntroduction & Philosophy

C*, sistem programlama alanına konumlanmış, LLVM tabanlı native kod üreten, pre-alpha aşamasında bir dildir. Amacı nettir: C/C++ ailesindeki düşük seviye kontrolü korumak, ama en pahalı hata sınıflarını dilin yüzeyinde görünür ve denetlenebilir kılmak.

C* is a pre-alpha systems programming language that produces native code through LLVM. Its goal is precise: keep the low-level control of the C/C++ family while making the most expensive classes of bugs visible and checkable on the language surface.

Çoğu modern dil belleği bir çalışma zamanının arkasına saklar. C* tam tersini yapar: pointer'ı, sahipliği ve nesne ömrünü saklamaz; bunları tip sisteminin birinci sınıf vatandaşı yapar. Bir T* paylaşılan bir işaretçidir, bir T^ taşınabilir tek sahipliktir, bir T& ödünç alınmış bir referanstır — ve derleyici bu üç kategorinin kurallarını ayrı ayrı uygular. Sonuç, "sihirli runtime kancası" yerine ABI düzeyinde görünür, incelenebilir lowering'dir.

Most modern languages hide memory behind a runtime. C* does the opposite: it does not hide pointers, ownership, or object lifetime; it makes them first-class citizens of the type system. A T* is a shared pointer, a T^ is movable single ownership, a T& is a borrowed reference — and the compiler enforces the rules of each category separately. The result is ABI-visible, inspectable lowering instead of "magic runtime hooks".

1.1Yedi Tasarım KararıSeven Design Decisions

Dilin tamamı şu yedi karardan türetilir. Bu kitaptaki her bölüm, bu kararlardan en az birinin somutlaşmış hâlidir:

The entire language derives from these seven decisions. Every chapter in this book is a concrete instance of at least one of them:

  • Bellek ve ömür açıktır. Hiçbir allocation, hiçbir yaşam döngüsü olayı programcıdan gizlenmez.
  • Memory and lifetime are explicit. No allocation and no lifecycle event is hidden from the programmer.
  • Pointer gizlenmez, denetlenir. Qualifier ve ownership kuralları pointer'ı yasaklamak yerine sınıflandırır.
  • Pointers are not hidden — they are governed. Qualifiers and ownership rules classify pointers instead of banning them.
  • struct gerçek layout sahibidir. Kalıtım (inheritance) yoktur; kompozisyon vardır.
  • struct owns a real layout. There is no inheritance; there is composition.
  • constructor allocation değildir. Var olan storage'ı initialize eder. destructor free değildir; nesne ömrünün kapanışıdır.
  • constructor is not allocation. It initializes existing storage. destructor is not free; it is the close of object lifetime.
  • new derleyiciye ayrılmıştır. Kullanıcı operator new yazamaz; allocation semantiği dilin kontrolündedir.
  • new is compiler-reserved. Users cannot write operator new; allocation semantics belong to the language.
  • Trait varsayılan olarak statiktir. Runtime dispatch yalnızca açık dynamic Trait&/dynamic Trait^ yüzeyiyle istenir.
  • Traits are static by default. Runtime dispatch is requested only through the explicit dynamic Trait&/dynamic Trait^ surface.
  • Proposal özellikler sessizce kabul edilmez. Henüz tamamlanmamış her yüzey kontrollü bir diagnostic üretir — asla sessiz yanlış davranış değil.
  • Proposal features are never silently accepted. Every unfinished surface produces a controlled diagnostic — never silent misbehavior.
note[DESIGN]: neden bu felsefe?why this philosophy?

Sistem yazılımında en pahalı hatalar bellek hatalarıdır: use-after-free, double-free, sahipsiz pointer, yanlış aliasing. C bu hataları görünmez kılar; garbage-collected diller ise maliyetli bir runtime ile satın alır. C*'ın konumu üçüncü yoldur: hatayı çalıştırmadan önce, derleyici diagnostiği olarak yüzeye çıkarmak. Bu yüzden dilin her kuralı — move-only ^, string literal kısıtları, drop-sonrası-kullanım hataları — aynı tek hedefe hizmet eder.

In systems software the most expensive bugs are memory bugs: use-after-free, double-free, unowned pointers, bad aliasing. C makes them invisible; garbage-collected languages buy them off with a costly runtime. C* takes the third road: surface the bug before execution, as a compiler diagnostic. Every rule in the language — move-only ^, string-literal restrictions, use-after-drop errors — serves that single goal.

Bu kitap iki katmanı kesin bir çizgiyle ayırır. Çalışan C*, examples/smoke/ ve examples/type_checker/ altındaki 250'den fazla test dosyasıyla doğrulanmış yüzeydir; bu kitaptaki ✓ pass etiketli her örnek bugün derlenir ve çalışır. This book separates two layers with a hard line. Working C* is the surface verified by more than 250 test files under examples/smoke/ and examples/type_checker/; every example tagged ✓ pass in this book compiles and runs today. The proposal layer is the future surface designed under examples/papers/; it is tagged ◆ proposal and mapped collectively in §22.

Kötü derlenen kod da bu kitabın birinci sınıf içeriğidir: ✗ diagnostic etiketli örnekler derleyicinin bilerek reddettiği programlardır. Bir dilin karakteri, neyi kabul ettiğinden çok neyi reddettiğinde okunur.

Ill-formed code is first-class content in this book too: examples tagged ✗ diagnostic are programs the compiler rejects on purpose. A language's character is read less in what it accepts than in what it refuses.

§02

Derleyici CMake ile inşa edilir ve beş aşamalı bir pipeline çalıştırır: lexer → parser → sembol analizi (pass0) → tip denetimi (pass1) → LLVM codegen. Çıktılar .ll, .s ve native executable olarak .cstar-out/ altına yazılır.

The compiler is built with CMake and runs a five-stage pipeline: lexer → parser → symbol analysis (pass0) → type checking (pass1) → LLVM codegen. Outputs land in .cstar-out/ as .ll, .s, and a native executable.

2.2Sonuç Modeli: Exit CodeThe Result Model: Exit Codes

C*'ta ret 7; konsola 7 yazdırmaz — process exit code'unu 7 yapar. Bu, dilin test altyapısının temelidir: her programın sonucu işletim sistemine verdiği exit status ile doğrulanır. PowerShell'de bu değer $LASTEXITCODE ile okunur. Üretilen programın sıfır dışı exit code'u derleyici hatası sayılmaz; runner bunu ayrı raporlar.

In C*, ret 7; does not print 7 — it sets the process exit code to 7. This is the foundation of the language's test infrastructure: every program's result is verified through the exit status it hands the operating system. In PowerShell that value is read via $LASTEXITCODE. A non-zero exit code from the generated program is not a compiler failure; the runner reports it separately.

Parsing stdout is brittle and platform-dependent; an exit code reads identically in every shell and every CI environment. Even before the language has a string type, this model delivers full test coverage — a deliberate, correct engineering decision for a pre-alpha compiler.

§03

Program YapısıProgram Structure

Bir C* dosyası top-level deklarasyonların dizisidir: global değişkenler, fonksiyonlar, struct/trait/enum tanımları ve include/import/export ifadeleri. Giriş noktası main'dir.

A C* file is a sequence of top-level declarations: global variables, functions, struct/trait/enum definitions, and include/import/export statements. The entry point is main.

3.1En Küçük Program ve :: İmzasıThe Smallest Program and the :: Signature

minimal.cstar✓ pass · exit 0
// expected: pass
// expected-exit: 0

main() :: int32 {
  ret 0;
}

Dönüş tipi :: ayracıyla imzanın sonuna yazılır. Bu, C'nin int main() önek stilinden bilinçli bir kopuştur: fonksiyonun adı ve parametreleri önce okunur, ürettiği tip sonra. Dönüş tipi hiç yazılmazsa fonksiyon void kabul edilir. Dönüş ifadesi ret'tir.

The return type is written after the signature, behind the :: separator. This is a deliberate break from C's int main() prefix style: the function's name and parameters read first, the type it produces reads last. If no return type is written, the function is treated as void. The return statement is ret.

void-return.cstar✓ pass
log_done() :: void {
  ret;
}
void-with-value.cstar✗ diagnostic
main() :: void {
  ret 0; // error: void function cannot return a value
}

3.2Yorumlar ve Satır DevamıComments and Line Continuation

Tek satır // ve çok satır /* ... */ yorumları desteklenir; lexer yorumları token akışına dahil etmez. İfadeler, parantez ve argüman bağlamlarında serbestçe yeni satıra taşınabilir — noktalı virgül sonlandırıcıdır, satır sonu değildir:

Single-line // and multi-line /* ... */ comments are supported; the lexer keeps comments out of the token stream. Expressions flow freely across lines inside parentheses and argument contexts — the semicolon is the terminator, not the line break:

newline-continuation.cstar✓ pass · exit 19
add(int32 a, int32 b, int32 c) :: int32 {
  ret a + b + c;
}

main() :: int32 {
  int32 matrix[2:3] = ((1, 2, 3),
                       (4, 5, 6));
  int32 arithmetic = 1 +
                     2 *
                     3;
  int32 call_value = add(1,
                         2,
                         3);
  ret arithmetic + call_value + matrix[1:
                                       2];
}

3.3Top-Level SıralamaTop-Level Ordering

Fonksiyonlar ileri (forward) çağrılabilir: derleyici pass'leri tüm fonksiyon imzalarını gövdelerden önce görür, bu yüzden tanım sırası çağrı sırasını kısıtlamaz. Global değişkenler, tip tanımları ve interop ifadeleri aynı dosyada serbestçe sıralanır.

Functions can be called forward: compiler passes see all function signatures before bodies, so definition order never constrains call order. Global variables, type definitions, and interop statements order freely within a file.

§04

Tip SistemiType System

C* statik ve açık tiplidir. Tip sistemi üç sınıftan oluşur: sabit genişlikli primitive'ler, mimariye bağlı alias'lar ve kullanıcı tanımlı tipler (struct, enum, trait). Tip çıkarımı yoktur; her deklarasyon tipini deklare eder.

C* is statically and explicitly typed. The type system has three classes: fixed-width primitives, architecture-dependent aliases, and user-defined types (struct, enum, trait). There is no type inference; every declaration declares its type.

4.2Tanımlı TiplerDefined Types

Kullanıcı tanımlı tipler üç yapıdan doğar ve her biri kendi bölümünde derinlemesine işlenir: struct gerçek bellek layout'u taşıyan değer tipidir (§14), enum açık temsilli skaler etikettir (§15), trait derleme zamanı davranış sözleşmesidir (§16).

User-defined types arise from three constructs, each covered in depth in its own chapter: struct is a value type carrying a real memory layout (§14), enum is a scalar tag with an explicit representation (§15), and trait is a compile-time behavioral contract (§16).

note[DESIGN]: neden tip çıkarımı yok?why no type inference?

Sistem kodunda tipin genişliği davranıştır: taşma sınırı, ABI genişliği, alignment ve register seçimi tipe bağlıdır. Açık tip, kodu okuyan kişiye bu bilgiyi arama yaptırmadan verir. C*'ın := token'ı bile tip çıkarımı değildir — o, ownership transferi için ayrılmıştır (§12.2). Dil bu ayrımı bilinçli korur: sözdizimsel kısalık, semantik görünürlüğe feda edilmez.

In systems code, type width is behavior: overflow boundaries, ABI width, alignment, and register selection all follow the type. An explicit type hands the reader that information without a lookup. Even C*'s := token is not type inference — it is reserved for ownership transfer (§12.2). The language protects this distinction deliberately: syntactic brevity is never traded against semantic visibility.

§05

Değişkenler ve AtamaVariables & Assignment

Deklarasyon biçimi her yerde aynıdır: Tip isim = başlangıç;. Değişkenler local, global veya modül-static olabilir; atama yüzeyi tam bir bileşik operatör ailesi taşır.

The declaration form is uniform everywhere: Type name = initializer;. Variables can be local, global, or module-static; the assignment surface carries a full family of compound operators.

5.1Local, Global ve Sıfır BaşlatmaLocals, Globals, and Zero-Init

variables.cstar✓ pass · exit 7
int32 global_count = 7;

main() :: int32 {
  int32 x = global_count;
  int32 y;          // başlatılmadı / uninitialized

  y = x;
  ret y;
}

Başlatılmamış local primitive değerler mevcut MVP'de zero-init davranışı alır: belirsiz değer okumak mümkün değildir. Bu karar ileride daha sıkı bir definite-assignment analizine dönüşecektir; her iki dünyada da sonuç aynıdır — rastgele bellek asla programın girdisi olmaz.

Uninitialized local primitives receive zero-init behavior in the current MVP: reading an indeterminate value is impossible. This decision will later tighten into a definite-assignment analysis; in both worlds the outcome is identical — random memory is never program input.

5.2Atama Operatör AilesiThe Assignment Operator Family

assignment-surface.cstar✓ pass
x = 1;
x += 2;   x -= 1;   x *= 3;   x /= 2;   x %= 5;
x <<= 1;  x >>= 1;
x &= 3;   x |= 4;   x ^= 2;

Bileşik atamalar skalerler üzerinde tanımlıdır ve okunacakları gibi davranırlar: x += 2, x = x + 2'nin tek değerlendirme garantili biçimidir.

Compound assignments are defined on scalars and behave exactly as they read: x += 2 is the single-evaluation form of x = x + 2.

5.3= Kopyalar, := Taşır= Copies, := Moves

C* iki atama token'ını anlamsal olarak ayırır. = değer kopyalamadır. := ise yalnızca unique pointer (^) için tanımlı move assignment'tır: sahipliği kaynaktan hedefe taşır, kaynağı geçersizleştirir. Primitive skaler üzerinde := kullanılmaz.

C* separates its two assignment tokens semantically. = is value copy. := is move assignment, defined only for unique pointers (^): it transfers ownership from source to target and invalidates the source. := is never used on primitive scalars.

move-assignment.cstar✓ pass · exit 9
main() :: int32 {
  int32 first = 3;
  int32 second = 9;
  int32^ target = ref first;
  int32^ source = ref second;

  target := source;      // sahiplik taşındı / ownership moved

  ret deref target;      // 9
}
note[DESIGN]: neden ayrı bir move token'ı?why a distinct move token?

C++'ta bir = ataması kopyalama mı taşıma mı yaptığını tipine, overload çözümüne ve value kategorisine göre gizlice seçer; okur bunu satırdan göremez. C* bu belirsizliği token düzeyinde kaldırır: = gördüğünüz her yerde kopya, := gördüğünüz her yerde sahiplik transferi vardır. Ownership akışı diff'te, code review'da ve grep'te görünür hale gelir.

In C++, whether an = assignment copies or moves is chosen silently by type, overload resolution, and value category; the reader cannot tell from the line. C* removes that ambiguity at the token level: wherever you see = there is a copy, wherever you see := there is an ownership transfer. Ownership flow becomes visible in diffs, code review, and grep.

§06

Operatörler ve İfadelerOperators & Expressions

İfade parser'ı sabit bir öncelik tablosuna dayanır ve C ailesinin tanıdık aritmetik, bitsel, karşılaştırma ve mantık yüzeyini taşır. Fark, kural setindedir: C*'ın operatörleri kategorilere karşı katıdır — pointer sıralanamaz, bool sıralanamaz, sayısal olmayan operand'a unary aritmetik uygulanamaz.

The expression parser rests on a fixed precedence table and carries the familiar arithmetic, bitwise, comparison, and logic surface of the C family. The difference lies in the rule set: C*'s operators are strict about categories — pointers cannot be ordered, bools cannot be ordered, and unary arithmetic never applies to non-numeric operands.

6.1Tam Yüzey Tek ProgramdaThe Full Surface in One Program

operator_precedence_stress.cstar✓ pass
main() :: int32 {
  int32 a = 5;
  int32 b = 3;
  int32 c = 2;

  int32 tick = 1;
  tick++;  ++tick;  tick--;  --tick;

  int32 arith = ((+a + b * c) - ((a - c) / b) + (a % c));
  int32 bitmix = ((5 & 3) | (8 ^ 2)) + ((16 >> 2) + (1 << 3));
  int32 inverted = ~0;
  int32 narrowed = cast<int32>(4.9) + (5 as int32);

  bool logic = !false && ((arith == 11) && (bitmix >= 23)) && !(a < b) &&
               ((a != c) || false);

  int32 selected = logic ? (arith + bitmix + narrowed - tick) : 100;
  int32* pointer = ref selected;
  int32 through_pointer = pointer ? deref pointer : 0;

  ret through_pointer;
}

6.2Unary Operatör KurallarıUnary Operator Rules

  • Unary + ve - yalnızca sayısal operand kabul eder. +pointer bir diagnostic'tir (CST2001).
  • Unary + and - accept only numeric operands. +pointer is a diagnostic (CST2001).
  • ! koşul-benzeri değerleri bool'a indirger.
  • ! lowers condition-like values to bool.
  • ~ yalnızca tamsayı veya flags enum üzerinde geçerlidir; ~1.5 diagnostic üretir.
  • ~ is valid only on integers or flags enums; ~1.5 produces a diagnostic.
  • ++/-- prefix ve postfix biçimde, statement düzeyinde skaler değişken mutasyonudur — overload edilemez (§16.3).
  • ++/--, prefix and postfix, are scalar variable mutation at statement level — they cannot be overloaded (§16.3).

6.3Karşılaştırma: Kategorilere SaygıComparison: Respecting Categories

Eşitlik (==, !=) tüm karşılaştırılabilir kategorilerde tanımlıdır. Sıralı karşılaştırma (<, >, <=, >=) ise yalnızca sayısal kategoride tanımlıdır:

Equality (==, !=) is defined across all comparable categories. Ordered comparison (<, >, <=, >=) is defined only in the numeric category:

bool-ordering.cstar✗ diagnostic
bool a = true;
bool b = false;
bool c = a > b;   // diagnostic: bool sıralanamaz / bool cannot be ordered
note[DESIGN]: neden pointer ve bool sıralanamaz?why can't pointers and bools be ordered?

İki pointer'ın < ilişkisi ancak aynı allocation içinde anlamlıdır; C bunun dışını undefined behavior yapar ama derleyici sizi uyarmaz. C* bu tuzağı kökten kapatır: pointer'lar yalnızca eşitlikle karşılaştırılır. bool için ise true > false gibi ifadeler neredeyse her zaman bir yazım hatasının veya C alışkanlığıyla yazılmış örtük int dönüşümünün izidir — dil bunu sayısal karşılaştırma saymayı reddeder ve niyeti açıkça sormanızı ister.

A < relation between two pointers is meaningful only within a single allocation; C makes anything else undefined behavior yet warns you never. C* closes that trap at the root: pointers compare by equality only. For bool, an expression like true > false is almost always the residue of a typo or a C-style implicit int habit — the language refuses to treat it as numeric comparison and demands you state the intent.

6.4Ternary İfadeThe Ternary Expression

  • Her iki dal değer üretmek zorundadır ve tipleri uyumlu olmalıdır.
  • Both branches must produce values, and their types must be compatible.
  • Mevcut MVP'de dallar yan etkisiz olmalıdır: fonksiyon çağrısı, atama veya allocation içeren dallar için if/else kullanılır. Ternary bir seçim ifadesidir, gizli bir kontrol akışı değildir.

Pointer koşulu ve deref birleşimi, null-korumalı okumanın deyimsel biçimidir:

§07

Cast SistemiThe Cast System

C* tip dönüşümünü iki güvenlik sınıfına böler: güvenli cast (cast<T>(expr) ve expr as T) yalnızca aynı kategori içinde dönüştürür; unsafe_cast<T> kategori sınırlarını geçer ve tüm sorumluluğu programcıya devreder.

C* splits type conversion into two safety classes: safe casts (cast<T>(expr) and expr as T) convert only within a category; unsafe_cast<T> crosses category boundaries and hands full responsibility to the programmer.

7.1Güvenli Cast: cast<T> ve asSafe Casts: cast<T> and as

safe-casts.cstar✓ pass
int32 a = cast<int32>(4.9);   // fonksiyon-benzeri biçim / function-like form → 4
int32 b = 5 as int32;         // infix biçim / infix form

İki biçim aynı semantiği taşır; cast<T> iç içe ifadelerde, as ise akıcı okunan tek dönüşümlerde tercih edilir. Sayısal daralma (float → int) açık cast ister; primitive genişleme bağlamları (ör. bool → int32 parametre kuralı, §8.2) dilin tanımladığı yerlerde örtük çalışır.

Both forms carry identical semantics; cast<T> reads better nested, as reads better as a single fluent conversion. Numeric narrowing (float → int) demands an explicit cast; primitive widening contexts (e.g. the bool → int32 parameter rule, §8.2) work implicitly where the language defines them.

7.2Kategori Sınırı: Value ↔ PointerThe Category Boundary: Value ↔ Pointer

Güvenli cast, value kategorisi ile pointer kategorisi arasındaki sınırı geçemez. Bir pointer'ı tamsayıya çevirmek adres aritmetiği dünyasına adım atmaktır ve bu adım asla kazara atılmaz:

A safe cast cannot cross the boundary between the value category and the pointer category. Turning a pointer into an integer is a step into address arithmetic, and that step is never taken by accident:

category-crossing.cstar✗ diagnostic
main() :: int32 {
  int32 value = 7;
  int32* p = ref value;
  int64 raw = cast<int64>(p);   // diagnostic: safe cast kategori geçemez
  ret 0;
}
unsafe-crossing.cstar✓ pass
main() :: int32 {
  int64 raw = 0;
  int32* p = unsafe_cast<int32*>(raw);   // açık niyet / explicit intent
  ret 0;
}
note[DESIGN]: neden "unsafe" kelimesi isimde?why is "unsafe" in the name?

unsafe_cast adı bilinçli olarak rahatsız edicidir. ABI, layout ve geçerlilik sorumluluğu bu satırda programcıya geçer — ve bu geçiş, kod tabanında grep unsafe_cast kadar basit bir sorguyla envanterlenebilir olmalıdır. C++'ın dört cast'lik ailesinin aksine C* ikili bir model seçer: güvenli ya da değil. Gri ton yoktur; denetim yüzeyi minimaldir.

The name unsafe_cast is deliberately uncomfortable. Responsibility for ABI, layout, and validity transfers to the programmer on that line — and that transfer must be inventoriable with a query as simple as grep unsafe_cast. Unlike C++'s four-cast family, C* chooses a binary model: safe or not. There is no gray; the audit surface is minimal.

§08

Fonksiyonlar top-level tanımlanır, ileri çağrılabilir ve imzaları isim(parametreler) :: DönüşTipi biçimini taşır. Parametre yazım sırası bile semantik taşır: C* declarator sırasını, sembolün örtük dönüşüm iznini deklare etmek için kullanır.

Functions are defined at top level, callable forward, and carry the signature form name(parameters) :: ReturnType. Even parameter word order carries semantics: C* uses declarator order to declare a symbol's implicit-conversion permission.

8.1Tanım, Çağrı ve Call StatementDefinition, Calls, and Call Statements

}

  ret;
}

Fonksiyon çağrısı bir ifade olarak initializer'da, argümanda ve dönüş değerinde kullanılır; tek başına bir satırda call statement olur. Forward çağrı serbesttir: derleyici pass'leri tüm imzaları gövdelerden önce çözümler.

A function call is an expression usable in initializers, arguments, and return values; alone on a line it becomes a call statement. Forward calls are free: compiler passes resolve all signatures before bodies.

8.2Parametre Cast Politikası: bool y vs y boolThe Parameter Cast Policy: bool y vs y bool

Bu, C*'a özgü bir mekanizmadır. Standart sıra Tip isim sembole normal örtük primitive dönüşüm izni verir. Ters sıra isim Tip ise aynı tipi taşır fakat sembolü scope içinde örtük cast'e kapatır:

This mechanism is unique to C*. The standard order Type name grants the symbol normal implicit primitive conversion. The reversed order name Type carries the same type but closes the symbol to implicit casts within its scope:

note[DESIGN]: neden declarator sırası bir kısıt?why is declarator order a constraint?

Bazı parametreler gerçekten sayı gibi davranmalıdır; bazıları ise — bir bayrak, bir kimlik, bir mod — aritmetiğe sızarsa bug'dır. C* bu niyeti ayrı bir keyword icat etmeden, mevcut sözdiziminin serbestlik alanına kodlar. Maliyeti sıfırdır, kazancı yerel bir tip disiplinidir: fonksiyon gövdesi, parametrenin sayısallaşmasını istemediğinizi derleyiciye söylemenin en kısa yoludur.

Some parameters genuinely behave as numbers; others — a flag, an identity, a mode — are bugs the moment they leak into arithmetic. C* encodes that intent into the freedom already present in its syntax, without inventing a keyword. The cost is zero; the payoff is local type discipline: the function body gets the shortest possible way to tell the compiler a parameter must never turn numeric.

note[ABI]: variadic sınırıthe variadic boundary

Variadic imza (...) yalnızca native import'ta tanımlıdır (§17.3). C* içinde variadic fonksiyon gövdesi yazılamaz; bu yüzey C ABI çağrısı içindir. Dil, kendi variadic modelini tip-güvenli bir tasarım hazır olana kadar açmaz.

Variadic signatures (...) exist only in native imports (§17.3). A variadic function body cannot be written in C*; that surface exists for C ABI calls. The language will not open its own variadic model until a type-safe design is ready.

8.3ÖzyinelemeRecursion

Fonksiyonlar kendi kendilerini çağırabilir. Özyineleme, normal call graph'ın parçasıdır: dönüş tipi açık yazılır, base case normal if ile kurulur ve recursive çağrı sıradan ifade gibi davranır.

Functions may call themselves. Recursion is part of the normal call graph: the return type is explicit, the base case is written with a regular if, and the recursive call behaves like any other expression.

recursive-fibonacci.cstar✓ pass · exit 55
fib(int32 n) :: int32 {
  if (n <= 1) {
    ret n;
  }

  ret fib(n - 1) + fib(n - 2);
}

main() :: int32 {
  ret fib(10);
}

Özyineleme struct layout özyinelemesiyle karıştırılmamalıdır: fonksiyon çağrısı runtime stack üzerinde ilerler; by-value kendini içeren struct alanı ise sonsuz boyut üreteceği için yasaktır (§14.1).

Recursion should not be confused with recursive struct layout: function calls progress on the runtime stack; a by-value struct field containing its own type would have infinite size and is rejected (§14.1).

Kontrol AkışıControl Flow

Kontrol akışı üç yapıya iner: if / elif / else zinciri, tek anahtar kelimelik loop ailesi ve enum'lara özel option dallanması (§15.3). while, for, do ayrımı yoktur — tek bir döngü formu üç semantiği taşır.

Control flow reduces to three constructs: the if / elif / else chain, the single-keyword loop family, and enum-specific option branching (§15.3). There is no while/for/do split — one loop form carries all three semantics.

9.1if / elif / else ve Koşul Dönüşümleriif / elif / else and Condition Conversions

elif tek token'dır; else if merdivenlerinin girinti sorununu kökten kaldırır. Koşul dönüşüm kuralları dört kategoride kesindir: bool doğrudan; tamsayı ve float sıfır değilse true; pointer null-benzeri değilse true.

elif is a single token; it eliminates the indentation problem of else if ladders at the root. Condition conversion is exact across four categories: bool passes directly; integers and floats are true when non-zero; pointers are true when non-null-like.

break and continue are valid only inside loop bodies; use outside a loop produces a diagnostic. The range form [1, 4] is a closed interval — both endpoints are included.

note[DESIGN]: neden tek döngü anahtar kelimesi?why a single loop keyword?

C ailesinde for(;;), while, do-while üç ayrı gramerle aynı işi yapar ve for'un üç bölmeli başlığı off-by-one hatalarının klasik evidir. C* iterasyonun ne üzerinde döndüğünü tek forma taşır: koşul mu, aralık mı, koleksiyon mu — parantezin içi bunu söyler, gövde her zaman aynıdır. Okuyucunun ezberlemesi gereken tek gramer vardır.

In the C family, for(;;), while, and do-while do the same job with three grammars, and for's three-slot header is the classic home of off-by-one bugs. C* moves what the iteration runs over into a single form: condition, range, or collection — the parenthesis says which, and the body is always the same. The reader memorizes exactly one grammar.

9.3Döngü Scope'u ve ArgümanlarLoop Scope and Arguments

Döngü başlığında üretilen semboller yalnızca gövdede yaşar. Gövde içinde normal local gibi davranırlar: fonksiyona argüman verilebilir, dynamic trait method'una geçirilebilir ve aritmetikte kullanılabilir. Döngü kapandıktan sonra aynı sembol görünür değildir.

Symbols produced by the loop header live only inside the body. Within that body they behave like normal locals: they can be passed to functions, passed into dynamic trait methods, and used in arithmetic. Once the loop closes, the symbol is no longer visible.

C* arrays are fixed-size values with a real layout. Size is part of the type, initializer arity is checked at compile time, multi-dimensional access uses the : separator, and negative indices mean access from the end.

note[DESIGN]: neden [1:2], neden [1][2] değil?why [1:2] instead of [1][2]?

C'de m[1][2] aslında iki ayrı subscript ifadesidir ve m[1] tek başına bir ara değer (satır pointer'ı) üretir — bu, pointer decay ile birleşince kazara satır adresi sızdırmanın klasik yoludur. C*'ın m[1:2] biçimi tek atomik erişimdir: ara değer yoktur, decay yoktur, boyut sayısı ile index sayısı derlemede eşleşmek zorundadır. Erişim sözdizimi, deklarasyondaki [2:3] ile simetriktir.

In C, m[1][2] is really two separate subscript expressions, and m[1] alone produces an intermediate value (a row pointer) — combined with pointer decay, that is the classic way to leak a row address by accident. C*'s m[1:2] is one atomic access: no intermediate value, no decay, and the number of indices must match the number of dimensions at compile time. The access syntax mirrors the [2:3] in the declaration.

Sabit bir out-of-bounds index — negatif ya da pozitif — derlemede yakalanır: values[-3] iki elemanlı dizide diagnostic'tir. Dizi boyutu ise ayrı bir konudur: boyut daima derleme zamanı pozitif tamsayı literal'i olmalıdır.

A constant out-of-bounds index — negative or positive — is caught at compile time: values[-3] on a two-element array is a diagnostic. Array dimensions are a separate matter: a dimension must always be a compile-time positive integer literal.

The array size in a parameter is part of the signature: passing an array of a different size produces a diagnostic. C's silent "array parameters are really pointers" decay rule does not exist in C*. Large local arrays are also rejected at compile time to prevent stack overflow:

  readonly int32* view = ref value;   // salt-okunur akış / read-only path
  ret read_ref(ref value) + deref view - limit;
}

readonly hedefe yazma girişimi diagnostic üretir. Bu dört isim, API imzalarını kendi kendini belgeler kılar: constptr gören çağıran "adres sabit, veri değişebilir" sözleşmesini imzadan okur.

Writing through a readonly target produces a diagnostic. These four names make API signatures self-documenting: a caller who sees constptr reads the contract "address fixed, data mutable" straight from the signature.

§12

Ownership ModeliThe Ownership Model

Ownership, C*'ın merkez tezidir: her kaynağın sahibi tipte yazılıdır. T* paylaşımı, T^ tekil sahipliği temsil eder; taşıma açık (move, :=), taşıma yasağı açık (nomove), taşınmış değerin yeniden kullanımı derleme hatasıdır.

Ownership is C*'s central thesis: every resource's owner is written in its type. T* expresses sharing, T^ expresses single ownership; transfer is explicit (move, :=), transfer-prohibition is explicit (nomove), and reusing a moved value is a compile error.

12.1Shared Pointer * ve strong_countThe Shared Pointer * and strong_count

Mevcut derleyici T* için strong count davranışını MVP düzeyinde takip eder; by-value fonksiyon argümanı retain/copy semantiğiyle test edilir. strong_count introspection builtin'i bu sayacı okur — model şeffaftır, kara kutu değildir.

The current compiler tracks strong-count behavior for T* at MVP level; by-value function arguments are tested with retain/copy semantics. The strong_count introspection builtin reads that counter — the model is transparent, not a black box.

12.2Unique Pointer ^: Kopya Yasak, Taşıma AçıkThe Unique Pointer ^: Copy Forbidden, Move Explicit

Kurallar kesindir: ^ kopyalanamaz; fonksiyona ancak move ile girer; değişkenler arasında ancak := ile taşınır (§5.3); fonksiyondan move ile döndürülür. Taşındıktan sonra kaynak semboller ölüdür — yeniden kullanım diagnostic üretir. Use-after-move, C*'ta bir runtime sürprizi değil bir derleme hatasıdır.

The rules are exact: ^ cannot be copied; it enters a function only via move; it transfers between variables only via := (§5.3); it returns from a function via move. After a move, the source symbol is dead — reuse produces a diagnostic. Use-after-move is a compile error in C*, never a runtime surprise.

12.3nomove: Taşımayı Sınırda Durdurnomove: Stopping Transfer at the Boundary

}

nomove, unique parametrenin fonksiyon gövdesinden dışarı taşınmasını yasaklayan bir ownership-flow modifier'dır — bir protocol state değildir. "Bu fonksiyon kaynağına bakar ama onu asla kaçırmaz" sözleşmesini imzaya yazar.

The nomove qualifier is an ownership-flow modifier that forbids the unique parameter from being moved out of the function body — it is not a protocol state. It writes the contract "this function inspects the resource but never abducts it" into the signature.

note[DESIGN]: iki pointer'lı bir dünya neden daha iyi?why is a two-pointer world better?

C'de tek pointer tipi beş ayrı rolü sırtlanır: sahip, ödünç alan, paylaşan, dizi, out-parametre. Hangi rolün geçerli olduğu dokümantasyonda yaşar ve dokümantasyon derlenmez. C* rolleri tipe böler: ^ gören her okur "tek sahip, taşınır" bilir; * gören "paylaşılan" bilir; & gören "ödünç" bilir. Double-free, tanımı gereği imkânsızlaşır — iki sahip aynı anda var olamaz çünkü tip sistemi ikinci kopyayı üretmeyi reddeder.

In C, one pointer type shoulders five roles: owner, borrower, sharer, array, out-parameter. Which role applies lives in documentation, and documentation does not compile. C* splits the roles into types: every reader of ^ knows "single owner, movable"; of *, "shared"; of &, "borrowed". Double-free becomes impossible by definition — two owners cannot coexist because the type system refuses to mint the second copy.

char* yasak, çünkü literal storage mutable değildir: yazma girişimi C'de segfault'tur, C*'ta derleme hatasıdır. char^ yasak, çünkü ^ tekil sahiplik iddiasıdır; literal ise derleyici/global storage'a aittir — sahiplenilemez, dolayısıyla drop edilemez. constptr char* yasak, çünkü constptr pointer'ın adresini sabitler ama hedefi immutable yapmaz; literal'in ihtiyacı hedef sabitliğidir. The correct types are precisely the two that tell the truth: const char* and readonly char*.

For C ABI compatibility, char* string pointers and void* opaque native handles lower as raw pointers; C*'s managed/shared pointer model is preserved for typed object pointers. When the real string type and allocator-backed dynamic buffers arrive (§22), this chapter will grow — today's model is not a stopgap, it is honest.

§14

Struct ve Yaşam DöngüsüStructs & Lifecycle

struct, C*'ın tek veri soyutlamasıdır ve gerçek bellek layout'u taşıyan bir değer tipidir. Kalıtım yoktur; davranış trait'lerle (§16), yapı kompozisyonla kurulur. Yaşam döngüsü dört ayrık kavramla yönetilir: constructor başlatır, destructor kapatır, new ayırır, drop erken serbest bırakır.

struct is C*'s single data abstraction and a value type carrying a real memory layout. There is no inheritance; behavior comes from traits (§16), structure from composition. Lifecycle is governed by four distinct concepts: constructor initializes, destructor closes, new allocates, drop releases early.

note[DESIGN]: neden dört ayrık yaşam döngüsü kavramı?why four distinct lifecycle concepts?

C++'ta new hem ayırır hem başlatır; destructor hem kapanıştır hem delete'in yarısıdır. Bu birleşme, placement-new, çift yıkım ve sızıntı hatalarının grameridir. C* dört kavramı dört ayrı kelimeye böler: ayırma (new) ↔ başlatma (constructor) ↔ kapanış (destructor) ↔ serbest bırakma anı (drop veya scope). Her kavram tek başına test edilebilir, tek başına okunabilir — stack'te constructor'lı bir değer allocation'sız yaşar, bu tasarımın kanıtıdır.

In C++, new both allocates and initializes; the destructor is both close-of-life and half of delete. That conflation is the grammar behind placement-new bugs, double destruction, and leaks. C* splits four concepts into four words: allocation (new) ↔ initialization (constructor) ↔ closing (destructor) ↔ the release moment (drop or scope). Each concept is testable and readable alone — a constructed value living on the stack without any allocation is this design's proof.

§15

Enum, Flags Enum ve OptionEnums, Flags Enums & Option

Enum'lar açık temsilli skaler etiketlerdir: her enum, bellekteki genişliğini : uint8 gibi deklare etmek zorundadır. Bit bayrağı semantiği ayrı bir tip sınıfıdır (flags enum) ve enum üzerinde dallanma, exhaustiveness denetimli option yapısıyla yapılır.

Enums are scalar tags with explicit representation: every enum must declare its in-memory width, as in : uint8. Bit-flag semantics is a separate type class (flags enum), and branching over enums uses the exhaustiveness-checked option construct.

15.1Skaler EnumScalar Enums

scalar-enum.cstar✓ pass · exit 1
enum Color : uint8 {
  Red,
  Green,
  Blue = 7,
}

main() :: int32 {
  Color color = Color.Green;
  if (color == Color.Green) {
    ret 1;
  }
  ret 0;
}
  • Explicit repr zorunludur: enum Color { ... } tek başına geçersizdir. Enum'un ABI genişliği asla platforma veya derleyici keyfine kalmaz.
  • An explicit repr is mandatory: bare enum Color { ... } is invalid. An enum's ABI width is never left to platform or compiler whim.
  • Üye erişimi daima nitelenmiş yazılır: Color.Green. Çıplak Green scope'a sızmaz — enum üyeleri global isim alanını kirletmez.
  • Member access is always qualified: Color.Green. Bare Green never leaks into scope — enum members do not pollute the global namespace.
  • Bilinmeyen üye diagnostic'tir; farklı enum tipleri birbiriyle karşılaştırılamaz ve karıştırılamaz.
  • An unknown member is a diagnostic; distinct enum types cannot be compared or mixed with one another.
  • Skaler enum üzerinde bitwise operatörler yasaktır — bu, flags enum'un işidir.
  • Bitwise operators on scalar enums are forbidden — that is the job of flags enums.

15.2Flags Enum: Power-of-Two DisipliniFlags Enums: The Power-of-Two Discipline

flags-enum.cstar✓ pass · exit 1
flags enum FileMode : uint32 {
  Read = 1,
  Write = 2,
  Append = 4,
}

main() :: int32 {
  FileMode mode = FileMode.Read | FileMode.Write;
  FileMode inverted = ~mode;

  if ((mode & FileMode.Write) == FileMode.Write) {

  }

  ret 0;

Flags enum kuralları: üye değerleri açık yazılmak zorundadır ve her değer 0 ya da power-of-two olmalıdır. |, &, ^, ~ yalnızca flags enum'da tanımlıdır. Derleyici Read = 3 gibi çakışan bit desenlerini reddeder — bir bayrak kümesinin cebiri, tanım anında doğrulanır.

Flags enum rules: member values must be explicit, and every value must be 0 or a power of two. |, &, ^, ~ are defined only on flags enums. The compiler rejects overlapping bit patterns like Read = 3 — the algebra of a flag set is verified at definition time.

15.3option: Exhaustive Dallanmaoption: Exhaustive Branching

option.cstar✓ pass
enum TokenKind : uint8 {
  Identifier,
  Integer,
  End,
}

score(TokenKind kind) :: int32 {
  option (kind) {
    TokenKind.Identifier: {
      ret 1;
    },
    TokenKind.Integer: {

option, C'nin switch'inden üç kritik noktada ayrılır: fallthrough yoktur — her dal kendi bloğudur; exhaustiveness denetlenir — tüm üyeler ya kapsanır ya _ ile açıkça devredilir; çift dal ve tip uyuşmazlığı diagnostic üretir. break yazmayı unutmak diye bir hata sınıfı C*'ta mevcut değildir.

The "forgot to write break" class of bug does not exist in C*.

note[DESIGN]: neden zorunlu repr?why a mandatory repr?

C'de bir enum'un boyutu implementation-defined'dır; aynı header iki derleyicide iki farklı ABI üretebilir. Ağ paketleri, dosya formatları ve FFI yapıları için bu kabul edilemez bir belirsizliktir. C* enum'u baştan bir layout sözleşmesi olarak tanımlar: : uint8 yazan, "bu etiket telde tam bir bayttır" demiş olur ve derleyici bu sözü her platformda tutar. Serileştirme kodu enum'u doğrudan cast'leyebilir çünkü genişlik tipin adında yazılıdır.

In C, an enum's size is implementation-defined; the same header can produce two different ABIs under two compilers. For network packets, file formats, and FFI structs that ambiguity is unacceptable. C* defines the enum as a layout contract from the start: writing : uint8 declares "this tag is exactly one byte on the wire," and the compiler keeps that promise on every platform. Serialization code can cast the enum directly because its width is written in the type's name.

§16

Trait ve Value OperatörlerTraits & Value Operators

Trait, derleme zamanında doğrulanan bir davranış sözleşmesidir. struct T with Trait yazımı statik conformance denetimi başlatır: eksik method diagnostic'tir. Trait alan enjekte etmez, runtime dispatch üretmez, kalıtım kurmaz — yalnızca kanıtlar.

A trait is a behavioral contract verified at compile time. Writing struct T with Trait triggers a static conformance check: a missing method is a diagnostic. A trait injects no fields, generates no runtime dispatch, and builds no inheritance — it only proves.

16.1Statik ConformanceStatic Conformance

trait-conformance.cstar✓ pass · exit 7
trait Measurable {
  value() :: int32;
}

struct Meter with Measurable {
  int32 reading;

  constructor(int32 value) {
    self.reading = value;
  }

  value() :: int32 {
    ret self.reading;
  }
}

main() :: int32 {
  Meter meter = Meter(7);
  ret meter.value();
}

Conformance beyanı tanımın üstündedir ve derleyici sözleşmenin her method'unu imza düzeyinde eşleştirir. Varsayılan çağrı dünyası vtable'sızdır: meter.value() doğrudan concrete method'a iner. Runtime dispatch istenirse dynamic yüzeyi ayrıca yazılır (§16.2).

The conformance declaration sits on the definition, and the compiler matches every method of the contract at the signature level. The default call world is vtable-free: meter.value() lowers directly to the concrete method. Runtime dispatch is requested separately through the dynamic surface (§16.2).

16.2Dynamic Trait Object'leriDynamic Trait Objects

dynamic Trait, statik conformance kanıtını runtime dispatch'e çeviren açık yüzeydir. Tipteki ownership işareti korunur: dynamic Writer& ödünç alınmış trait object'tir, dynamic Writer^ heap sahipliğini taşıyan trait object'tir. dynamic kelimesi maliyetin ve vtable geçişinin kodda görünür kalmasını sağlar.

dynamic Trait is the explicit surface that turns a static conformance proof into runtime dispatch. The ownership sigil is preserved: dynamic Writer& is a borrowed trait object, while dynamic Writer^ carries heap ownership. The dynamic keyword keeps the cost and vtable crossing visible in code.

dynamic-trait-borrowed.cstar✓ pass
trait Writer {
  write(const char* data) :: void;
}

struct ConsoleWriter with Writer {
  constructor() {
    ret;
  }

  write(const char* data) :: void {
    print(data);
    ret;
  }
}

log(dynamic Writer& writer) :: void {
  writer.write("dynamic writer dispatch\n");
  ret;
}

main() :: int32 {
  ConsoleWriter writer = ConsoleWriter();
  dynamic Writer& erased = dynamic ref writer as Writer;
  log(erased);
  ret 0;
}

Borrowed dynamic trait object kaynak nesnenin ömrünü sahiplenmez; yalnızca concrete değere ve o concrete tip için üretilmiş vtable'a bakar. Eksik method, yanlış trait adı veya trait'i implement etmeyen concrete tip diagnostic üretir.

A borrowed dynamic trait object does not own the source object's lifetime; it points at the concrete value and the vtable generated for that concrete type. A missing method, wrong trait name, or concrete type that does not implement the trait produces a diagnostic.

dynamic-trait-multi-method.cstar✓ pass · exit 32
trait Accumulator {
  add(int32 value) :: int32;
  current() :: int32;
}

struct BoxCounter with Accumulator {
  int32 total;

  constructor(int32 initial) {
    self.total = initial;
    ret;
  }

  add(int32 value) :: int32 {
    self.total += value;
    ret self.total;
  }

  current() :: int32 {
    ret self.total;
  }
}

run(dynamic Accumulator& acc) :: int32 {
  int32 a = acc.add(4);
  int32 b = acc.add(7);
  ret a + b + acc.current();
}

main() :: int32 {
  BoxCounter counter = BoxCounter(2);
  dynamic Accumulator& acc = dynamic ref counter as Accumulator;
  ret run(acc);
}

Dynamic dispatch normal ifade sistemine dahildir: method dönüşleri değişkene alınabilir, aritmetiğe katılabilir, döngü içinde çağrılabilir ve recursive fonksiyonlarla birlikte kullanılabilir.

Dynamic dispatch participates in the normal expression system: method returns can be stored in variables, used in arithmetic, called inside loops, and composed with recursive functions.

dynamic-trait-owned.cstar✓ pass · exit 17

trait Gauge {
  read() :: int32;

struct HeapGauge with Gauge {
  int32 value;

  constructor(int32 value) {
    self.value = value;
    ret;
  }

  destructor() {
    destroyed = self.value;
    ret;
  }

  read() :: int32 {
    ret self.value;
  }
}

make_and_read() :: int32 {
  HeapGauge^ gauge = new HeapGauge(17);
  dynamic Gauge^ erased = dynamic move gauge as Gauge;

  int32 value = make_and_read();
  ret value + destroyed - 17;

Diagnostic, C*'ta bir yan ürün değil, tasarlanmış bir çıktı formatıdır: dosya, satır, sütun, severity, kod ve caret işaretçisi. Hedef üçlüdür — crash değil kontrollü hata, hata tam kaynağında, proposal özellikler sessiz değil sesli.

In C*, a diagnostic is not a by-product but a designed output format: file, line, column, severity, code, and a caret marker. The goal is threefold — controlled errors instead of crashes, errors at their exact source, and proposal features loud, never silent.

Diagnostic yazımının üç ilkesi mutlaktır: assert veya crash yerine kontrollü teşhis; hata mesajı yanlış satırda değil, hatayı üreten ifadenin/sembolün üstünde; proposal özelliği sessizce parse edilmez — "implemented değil" diyen açık bir teşhis üretir. Three principles of diagnostic authoring are absolute: controlled diagnosis instead of asserts or crashes; the message lands on the expression/symbol that produced the error, never the wrong line; and proposal features are never silently parsed — they emit an explicit "not implemented" diagnosis. The 110+ files of the type-checker suite are this contract's regression shield.

Proposal HaritasıThe Proposal Map

Bu bölüm geleceğin tasarımını özetler. Buradaki kodlar examples/papers/*.cstar altında tutulur ve bugünkü derleyicide kontrollü diagnostic üretir — sessiz kabul asla. Her başlık, çalışan dilin hangi boşluğunu kapatacağını deklare eder.

This chapter summarizes the future design. Its code lives under examples/papers/*.cstar and produces controlled diagnostics in today's compiler — never silent acceptance. Each heading declares which gap in the working language it will close.

22.1Nullability: Non-Null VarsayılanNullability: Non-Null by Default

Plan kesindir: T*, T^, T& varsayılan olarak non-null olur. Nullable pointer açık ? ister (T*?, T^?); T& asla nullable olmaz. nil yalnızca nullable bağlamda legal olur ve if (p) koşulu true dalında pointer'ı non-null'a daraltır (flow narrowing):

The plan is exact: T*, T^, T& become non-null by default. A nullable pointer requires an explicit ? (T*?, T^?); T& is never nullable. nil becomes legal only in nullable context, and an if (p) condition narrows the pointer to non-null in the true branch (flow narrowing):

nullability.cstar◆ proposal
read_nullable(int32*? maybe) :: int32 {
  if (maybe) {
    ret deref maybe;   // burada maybe: int32* (non-null'a daraltıldı / narrowed)
  }

  ret 0;
}

22.2Allocator ve new?Allocators and new?

Allocation yüzeyi dört forma ayrılır: new Type(args) varsayılan allocator; new(allocator) Type(args) açık allocator; shared new paylaşımlı sahiplik; new? başarısız olabilen (fallible) allocation. Infallible new asla nil döndürmez — başarısızlık ihtimali varsa bu, tipte ? olarak görünmek zorundadır. Primitive allocation da legal kabul edilir:

The allocation surface splits into four forms: new Type(args) with the default allocator; new(allocator) Type(args) with an explicit one; shared new for shared ownership; new? for fallible allocation. Infallible new never returns nil — if failure is possible, it must appear as ? in the type. Primitive allocation is also legal:

allocator.cstar◆ proposal
Meter^ owned = new(arena) Meter(7);
Meter^? maybe = new?(arena) Meter(7);
int32^ owned_value = new(arena) int32(7);

22.3Dynamic ProtocolDynamic Protocols

Static/provable protocol ve scope_exit cleanup edge'leri artık çalışan yüzeydedir (§16.5). Proposal alanında kalan parça, runtime state tag isteyen dynamic protocol ve .= dynamic/provability-gap transition lowering'idir. Bu lowering gizli hook table veya hashmap kullanmayacak; --show-desugar ile görülebilir switch/check çağrılarına inecektir:

Static/provable protocols and scope_exit cleanup edges are now part of the working surface (§16.5). The remaining proposal area is dynamic protocol for runtime state tags and .= dynamic/provability-gap transition lowering. This lowering will not use hidden hook tables or hashmaps; --show-desugar must expose the generated switch/check calls:

dynamic-protocol.cstar◆ proposal
dynamic protocol ConnectionState for Connection {
  state idle, sending, failed;
  default idle;

  idle    -> sending :: send();
  sending -> idle    :: ack();
  sending -> failed  :: timeout();
}

handle(idle Connection^ conn) :: void {
  conn .= try_send();        // runtime tag check + visible desugar
  if (conn is failed) {
    ret;
  }
}

Bu tasarımda da ilke değişmez: ownership işareti tipte görünür kalır, dispatch maliyeti dynamic kelimesiyle açık yazılır.

22.6Reflection YüzeyiThe Reflection Surface

Attribute gövdelerinin (§18.3) hedef sözlüğü şu ilk yüzeyle tanımlanır: name($item), fields($item), methods($item), has_attribute(item, Name), attribute_args(item, Name), sizeof(Type), alignof(Type). Hepsi derleme zamanı sorgularıdır; runtime reflection maliyeti yoktur.

The target vocabulary of attribute bodies (§18.3) is defined by this initial surface: name($item), fields($item), methods($item), has_attribute(item, Name), attribute_args(item, Name), sizeof(Type), alignof(Type). All are compile-time queries; there is no runtime reflection cost.

22.7Async ve ConcurrencyAsync & Concurrency

Concurrency modeli, ownership modelinin doğrudan uzantısıdır: async task/thread sınır etkisidir, await bir suspension ve ownership checkpoint'idir. Send/Sync marker capability'lerdir. Sınır kuralları kesindir — T^ task sınırını yalnızca açık move ile geçer; T* retain/copy veya açık move ile geçer; ödünç & lifetime kanıtı ister:

The concurrency model is a direct extension of the ownership model: async is a task/thread boundary effect, await is a suspension and an ownership checkpoint. Send/Sync are marker capabilities. The boundary rules are exact — T^ crosses a task boundary only via explicit move; T* crosses via retain/copy or explicit move; a borrowed & demands a lifetime proof:

concurrency.cstar◆ proposal
consume_on_worker(int32^ owned) async :: int32 {
  ret deref owned;
}

main_task() async :: int32 {
  int32 value = 7;
  int32^ owned = ref value;
  ret await consume_on_worker(move owned);
}
note[ROADMAP]: değişmeyen ilkethe invariant principle

Haritadaki her özellik aynı üç filtreye tabidir: (1) ABI-visible lowering — sihir yok; (2) açık niyet — ?, dynamic, move, new? gibi maliyet işaretleri tipte/sözdiziminde görünür; (3) tamamlanana kadar kontrollü diagnostic. Dil büyürken karakteri sabittir.

Every feature on the map passes the same three filters: (1) ABI-visible lowering — no magic; (2) explicit intent — cost markers like ?, dynamic, move, new? appear in the type or syntax; (3) controlled diagnostics until complete. The language grows; its character does not move.
